# Harness Inventory

> The central map of the Agentic OS **harness**: the machinery around the model
> that makes agentic work reliable. It names each component, where it lives, what
> it owns, whether it is static or dynamic, manual or automated — and makes the
> gaps visible.

The model is not the whole agent:

```
Agent = Model + Harness
```

The **model** generates. The **harness** is everything else that makes the
generation reliable: static context, dynamic context, skills, tools, memory,
workflows, evals, guardrails, the execution environment, observability, and the
hooks that don't exist yet. This page inventories the harness as it stands today.
The schema in [`AGENTS.md`](../../AGENTS.md) still wins on any conflict.

## Why the harness matters

- **The model is swappable; the harness is the durable asset.** Prompts, skills,
  memory, conventions, and evals outlive any single model version. Investing in the
  harness compounds; tuning a single prompt does not.
- **Reliability lives in the harness, not the model.** Verification bars, integrity
  rules, placement discipline, and eval cases are what stop fabrication and drift —
  the model alone will not.
- **You can only improve what you can see.** An explicit inventory exposes what is
  defined-and-active vs. manual vs. not-built — so gaps get closed deliberately
  instead of discovered by failure.

This page is the umbrella over its three siblings: [Task Modes](task-modes.md) (how
much discipline a task needs), [Evals](evals.md) (did it meet the bar), and
[Context Engineering](context-engineering.md) (where the knowledge belongs). Each of
those is one harness component; this page shows how they fit the whole.

## Harness map

The flow a non-trivial task moves through, and which component owns each step:

```
User task
  → classify task mode            (Task Modes)
  → choose dynamic context         (Context Engineering)
  → load relevant skill / workflow / project memory   (Skills, Workflows, Projects)
  → perform work                   (Model + Tools)
  → verify                         (Verification: performed vs. recommended; Evals after the fact)
  → record eval / memory / doc updates if needed       (Raw → Wiki, Eval results)
  → update index / log if needed   (index.md, log.md)
  → commit explicitly              (Git boundary)
```

Not every task touches every step — an `exploration` may stop at "perform work,"
a `memory-update` centers on the record step — but the order is the spine.

## Current maturity

- **Defined and active:** Task Modes, Context Engineering policy, Skills, Workflow
  pages, Project memory, Raw layer, Examples, Index, Log, Guardrails/integrity rules,
  Git boundary, Tools.
- **Scripted (report-only):** the deterministic wiki lint (`scripts/wiki-lint.py`,
  2026-07-02) — links, frontmatter, sources, index coverage, log format/order/size,
  capture `Status:`, leaked markers; expanded 2026-07-03 with eval-result file
  format, skill frontmatter, harness drift (warning-level: script refs + the
  Skills-row list checked against the repo), and stray temp/generated/scratchpad
  candidates (warning-level). Invoked manually (via `/aos-wiki-lint` or
  pre-commit); it reports, it does not block. Since it is load-bearing (pre-commit,
  memory-health cadence), it carries its own regression suite: a `--root` seam plus
  clean/violating fixture pairs per check under `scripts/tests/` (stdlib `unittest`,
  14 tests — the linter had been untested until 2026-07-02).
- **Scripted sync check:** `scripts/sync-skills.py --check` runs in pre-commit when
  any skill surface changed and in CI for every push/PR; drift is treated as a
  pre-commit block because it means one model surface is stale.
- **Defined but manual:** Evals (no runner), Eval result logs, Guardrail hooks
  (checklist contract, no runner), AI code review (checklist, no runner), the
  memory-health cadence, user/session capture.
- **Active manual convention:** Verification conventions — the performed-vs-recommended
  split now has a [central page](verification.md), is wired into the `post-edit` /
  `pre-memory-update` / `pre-commit` hooks and an eval, and is applied on every result;
  enforcement is still by hand and review, not tooling. **Memory quality** is the same
  shape — a [central page](memory-quality.md) setting category authority, raw→curated
  promotion, and stale marking, wired into `pre-memory-update` and an eval; manual.
- **Partial / implicit:** Observability — present only through `log.md`, eval result
  files, and the Obsidian dashboard HUD; no structured tracing or metrics.
- **Report-only assistants (2026-07-03):** `/aos-task-mode`, `/aos-verify-block`,
  `/aos-hook`, `/aos-pre-commit`, and `/aos-eval` — the formerly deferred assistant skills,
  built once the manual contracts had proven their shape. Each walks its
  contract page and emits the standard block; **advisory only** — verdicts are
  reports, judgment-eval passes need user confirmation, nothing is mechanically
  gated. **Cold-invocation tested 2026-07-03 — all 5.** The first two
  (`/aos-task-mode`, `/aos-pre-commit`) were invoked by name by a fresh no-context
  subagent and returned the exact contract block (hotfix 7-field skeleton with the
  escalation check; the `pre-commit` `Hook` block that correctly did *not* claim the
  lint ran); the remaining three were cold-tested the same day — `/aos-verify-block`
  split performed/recommended with only the actually-run lint as performed,
  `/aos-hook pre-edit` emitted its `PASS` block, and `/aos-eval` drafted a **FAIL**
  and **refused to write a results file** (the judgment-guard held). All runs edited
  no files and performed no task — evidence the invocation contract holds; still not
  proof a cold agent *discovers* them unprompted (that is a separate eval, below).
- **Flow wrappers (2026-07-03):** `/aos-plan` and `/aos-implement` — thin
  orchestration skills over the existing contracts. `/aos-plan` never edits;
  `/aos-implement` edits and applies mode classification, context loading,
  required hooks, verification split, and diff review internally, surfacing only
  WARN/BLOCK hook findings plus final verification and risks.
- **Not implemented yet:** Model routing, and *executable* hook automation (git /
  Claude Code hooks, a runner) — both deferred (a wrong mechanical gate blocks
  real work). The guardrail-hook *definitions* and their report-only assistant
  now exist; only blocking automation stays deferred.
- **Model-agnostic skill migration (2026-07-03):** `skills/aos-*` is now the
  canonical skill source. Claude and Codex skill surfaces are generated mirrors,
  and `scripts/sync-skills.py --check` plus `scripts/wiki-lint.py` catch drift.

## Do not confuse

- **`AGENTS.md` is not the harness** — it is *one* static-context component. The
  harness is the whole system around the model.
- **Skills are not global memory** — they are dynamic procedural context, loaded only
  when their operation runs.
- **The project wiki is not raw memory** — it is curated, synthesized knowledge.
  `raw/` is the immutable evidence; `wiki/` is the synthesis.
- **Evals are not documentation** — they are behavioral checks with required and
  forbidden outputs. A workflow page describes; an eval verifies.
- **Logs are not proof of correctness** — `log.md` and eval result files are
  trace/history (what happened), not a guarantee the work was right.

## Known gaps / future candidates

- **Hooks** — the manual guardrail-hook *contract* is now defined
  ([guardrail-hooks.md](guardrail-hooks.md)); *executable* automation (git / Claude
  Code hooks, a runner) is still absent, every check run by hand today.
- **Observability** — only implicit (log, eval results, dashboard); no structured
  run tracing or metrics.
- **Model routing** — no per-task/per-mode model selection; modes could feed it later.
- ~~**Codex plugin installation docs / marketplace entry** — the plugin exists in
  the repo, but no marketplace entry or fresh-session install note has been captured.~~
  Resolved 2026-07-03: the personal marketplace entry points at
  `~/plugins/agentic-os` (a symlink to the repo plugin), `agentic-os@personal`
  is installed, and [agentic-os-usage](../agentic-os-usage.md) records the
  reinstall/new-thread note.
- **Session interaction store** — sessions are captured only as approved raw notes;
  there is no standing per-session record.
- **Eval automation** — deliberately absent (no runner). The `/aos-eval` **assistant**
  exists (2026-07-03) — it loads a case and drafts the comparison — but pass/fail
  on judgment evals still needs the user, and there is no automatic runner.
- **Cold-agent skill *discovery* eval** — **closed 2026-07-03.** The
  [cold-skill-discovery](evals/cases/cold-skill-discovery.md) case now covers what the
  invocation test did not: a cold agent must route plain-language tasks to the right
  `aos-*` skill with none named. Re-run it when the `aos-*` skill
  descriptions or the available-skills surface change.

These are candidates, not commitments. The standing posture is: keep the harness
lightweight and add automation only when the manual discipline proves its shape.
